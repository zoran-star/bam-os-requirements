the scaling question ✎

Pipeline Presets: Can We Scale It?

The ask: create new pipeline presets (not just free-trial), configure the agents and automations per stage in each preset, and have training flow per preset. Verdict: the foundation is real, three things are still welded shut, and there's a clean 4-phase path. Tech terms in black chips, plain English in green handwriting.

A ALREADY BUILT — stronger than it looks
the pipeline is already becoming data we own
🗺️ Stage registry
pipeline_stages table: one row per stage per academy, with a role + the GHL ids it maps to. A pipeline_provider flag flips reads from GHL to our own table.
The stages live in OUR database now, not just inside GoHighLevel.
🕸️ Transition graph
stage_transitions: an edge per row - (from_stage, trigger) → destination. LIVE in prod, seeded for BAM GTA (20 edges), and routeTransition() is already called by all 3 agents.
"When a lead does X in stage Y, send them to Z" is already data, not code.
⚙️ Automations engine
automations + steps / enrollments / jobs / events tables run the Ghosted and Nurture sequences portal-side.
Drip sequences are our system, not GHL workflows.
🧠 Brain-as-data + lessons
Each agent's prompt = 30 named sections overridable per academy. Lessons carry agent + scope, and shared ones are already tagged preset: free_trial.
The personality and training are editable content, not hardcode... mostly.

Even the "seed BAM GTA's flow" function is a primitive preset stamp - it just only knows ONE flow.

but three things are welded shut ↓↓
B WELDED SHUT — the 3 blockers
why a new preset can't exist today
🔒 Closed stage list
stage_role is a 7-value database enum: responded, interested, scheduled_trial, done_trial, nurture, won, unqualified. A CHECK constraint rejects anything else.
You literally cannot add an "Assessment Booked" stage - the database says no.
🔒 3 hardcoded agents
AGENT_SPECS = booking / confirm / closing, each welded to one stage AND with "get them to book a free trial" written into its default prompt text in code.
There's no way to say "make me a 4th agent" or "same agent, different mission."
🔒 No preset object
Nothing represents "a sales system" as a thing. The free-trial flow exists only as the union of hardcoded defaults + one seed function.
You can't create, name, copy, or assign a playbook - it isn't a noun in the system yet.
Training is the least-blocked layer. Lessons already carry agent + scope + a preset tag. What's missing: the readers don't FILTER by preset yet (harmless while only free-trial exists), and lessons key on the agent's hardcoded name instead of a template id.
the target shape ↓↓
C TARGET STATE — the entity map (updated with Zoran's decisions)
presets live in CODE · an academy's OFFER runs an instance
preset definitions in code, BAM-only
  • a versioned code registry (like prompt-structure.js today), not DB rows - only BAM edits it, in git
  • each preset = key ('free_trial', 'discovery_trial') + stage list (open role_keys) + transition graph + which worker runs each stage
  • no authoring UI to build or secure - a preset ships like a feature
＋
agent templates in code, reusable
  • key 'trial_booking', 'call_confirm', 'trial_confirm', 'closing'...
  • mission + default prompt sections + tool config (reply / book / schedule / suggest-lost)
  • shared across presets: preset #2 reuses trial_confirm + closing verbatim - only its booking mission is new
⇒ apply_preset(academy, OFFER, preset_key) stamps the instance
per-OFFER instance what actually runs
  • pipelines hang on the offer, not the academy - one academy can run free-trial youth training AND a discovery-call elite program side by side
  • pipeline_stages + stage_transitions (both live today) gain an offer_id; keys become (client, offer, role)
  • client_agents: (client × offer × agent_template) + mode (off / hawkeye / self_drive)
  • leads already route by offer: entry_points has offer_id today
→
customization envelope locked vs open
  • 🔒 stages + transitions + agent missions: locked to the preset (BAM re-stamps safely on preset upgrades)
  • 🔓 automations: timings + message copy tunable per instance
  • 🔓 agent training: lessons, brain facts, examples, mode - fully per academy
→
training scoped by template
  • lessons key on agent_template, not a hardcoded name
  • general lessons attach to the TEMPLATE - craft taught to trial_confirm helps EVERY preset that reuses trial_confirm, automatically
  • booking-for-trial vs booking-for-call are different templates, so their craft never cross-bleeds
  • /consolidate-lessons runs per template; intake mining feeds preset-specific onboarding questions
Why code-authored presets simplify everything: no template tables, no authoring UI, no permissions. A preset is reviewed and shipped like code (because it is code). And because academies can't restructure stages or transitions, BAM can safely RE-STAMP an improved preset onto existing instances - only automations + training are per-academy, and those survive the re-stamp.
how we get there without breaking BAM GTA ↓↓
D THE PATH — 4 phases, no big-bang rewrite
each phase ships value on its own
Phase 0done ✓
Foundation (already live). Stage registry, transition graph + router, automations engine, brain-as-data, preset-tagged lessons, Hawkeye training loop. This is why the answer is "3 pieces away," not "rewrite."
Phase 1done ✓ 2026-07-10
Unweld the stage list. SHIPPED + applied to prod. Migration 20260710170000_open_stage_role_vocabulary: dropped the closed 7-value CHECKs on pipeline_stages.role + opportunities.stage_role, converted stage_transitions' stage_role enum to text, dropped the enum, added a soft lowercase-snake_case format check. Verified in prod: a new role (discovery_call_booked) inserts, garbage rejects, all 119 existing rows intact, free-trial flow byte-identical. New stage names are now POSSIBLE - the database no longer rejects them. Dormant until a code preset uses one.
Phase 2the preset noun
Preset registry in code + per-OFFER instances. A versioned code registry defines presets (stages, transitions, worker per stage) and reusable agent templates; free_trial is written down from today's exact model. apply_preset(academy, offer, preset) replaces the one-off seeder. Schema side: pipeline_stages / stage_transitions / agent config gain offer_id (keys become client + offer + role). A second playbook is a code file; stamping an academy's offer with it is one call. ⚠ adding offer_id to live keyed tables (unique constraints) - additive but needs a careful backfill for BAM GTA
Phase 3generic workers
One engine, many agents. Collapse the 3 copy-pasted agent APIs into one detector/drafter parameterized by (academy, offer, stage, agent_template). Hawkeye's tabs render from the offer's actual stages instead of hardcoded booking/confirm/closing. A 4th agent (or a 5-stage preset) is a registry entry, not a new API file.
Phase 4train per template
Template-scoped training. agent_lessons.agent becomes the agent_template key (backfill today's 3), readers load general lessons by template, /consolidate-lessons runs per template, intake-gap mining feeds preset-specific onboarding questions. Craft taught to a shared template (trial_confirm, closing) upgrades EVERY preset that uses it - while different missions never cross-bleed.

Order matters: 1 before 2 (open roles first), 2 before 3 (the engine reads the registry), 4 can start any time after 2.

E DECISIONS — locked by Zoran, 2026-07-10
these shaped the map above
1 · Preset hangs on the OFFER ✓ One academy can run multiple offers, each with its own pipeline preset. Schema consequence: pipeline_stages / stage_transitions / agent config gain offer_id. Leads already route by offer via entry_points.
2 · Presets are authored by BAM, in CODE ✓ No authoring UI, no DB templates - a preset is a reviewed, versioned code file. Kills a whole class of build work and permissions questions.
3 · Per-instance tuning = automations + agent training ONLY ✓ Stage structure, transitions, and agent missions stay locked to the preset. Means BAM can safely re-stamp preset upgrades onto live academies - tuning survives, structure refreshes.
4 · Preset #2 outlined ✓ The discovery-call motion, sketched below. Reuses nurture + ghosted automations and the trial-confirm + closing agents as-is.
