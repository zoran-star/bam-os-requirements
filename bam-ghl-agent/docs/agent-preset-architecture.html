the scaling question ✎

Pipeline Presets: Can We Scale It?

The ask: create new pipeline presets (not just free-trial), configure the agents and automations per stage in each preset, and have training flow per preset. Verdict: the foundation is real, three things are still welded shut, and there's a clean 4-phase path. Tech terms in black chips, plain English in green handwriting.

A ALREADY BUILT — stronger than it looks
the pipeline is already becoming data we own
🗺️ Stage registry
pipeline_stages table: one row per stage per academy, with a role + the GHL ids it maps to. A pipeline_provider flag flips reads from GHL to our own table.
The stages live in OUR database now, not just inside GoHighLevel.
🕸️ Transition graph
stage_transitions: an edge per row - (from_stage, trigger) → destination. LIVE in prod, seeded for BAM GTA (20 edges), and routeTransition() is already called by all 3 agents.
"When a lead does X in stage Y, send them to Z" is already data, not code.
⚙️ Automations engine
automations + steps / enrollments / jobs / events tables run the Ghosted and Nurture sequences portal-side.
Drip sequences are our system, not GHL workflows.
🧠 Brain-as-data + lessons
Each agent's prompt = 30 named sections overridable per academy. Lessons carry agent + scope, and shared ones are already tagged preset: free_trial.
The personality and training are editable content, not hardcode... mostly.

Even the "seed BAM GTA's flow" function is a primitive preset stamp - it just only knows ONE flow.

but three things are welded shut ↓↓
B WELDED SHUT — the 3 blockers
why a new preset can't exist today
🔒 Closed stage list
stage_role is a 7-value database enum: responded, interested, scheduled_trial, done_trial, nurture, won, unqualified. A CHECK constraint rejects anything else.
You literally cannot add an "Assessment Booked" stage - the database says no.
🔒 3 hardcoded agents
AGENT_SPECS = booking / confirm / closing, each welded to one stage AND with "get them to book a free trial" written into its default prompt text in code.
There's no way to say "make me a 4th agent" or "same agent, different mission."
🔒 No preset object
Nothing represents "a sales system" as a thing. The free-trial flow exists only as the union of hardcoded defaults + one seed function.
You can't create, name, copy, or assign a playbook - it isn't a noun in the system yet.
Training is the least-blocked layer. Lessons already carry agent + scope + a preset tag. What's missing: the readers don't FILTER by preset yet (harmless while only free-trial exists), and lessons key on the agent's hardcoded name instead of a template id.
the target shape ↓↓
C TARGET STATE — the entity map
preset = template · academy = an instance of it
pipeline_presets the playbook
  • key 'free_trial', 'paid_assessment'...
  • name + description + status
  • BAM-authored template, owned globally
→
preset_stages its columns
  • role_key open text (not an enum!)
  • position, label, is_terminal
  • worker_kind agent | automation | human
  • → which agent_template or automation works it
→
preset_transitions its arrows
  • (from_role, trigger) → to_role/terminal
  • the default edge graph for this playbook
  • same shape as today's live table
＋
agent_templates the workers
  • key 'ft_booking', 'pa_closer'...
  • mission + default prompt sections (moves the free-trial prose OUT of code into seeded data)
  • tool config: reply / book / schedule / suggest-lost
  • today's 3 agents become the free_trial preset's seeded rows
⇒apply_preset(academy)
per-academy instance the copy they run
  • pipeline_stages rows stamped from preset_stages (role becomes open text)
  • stage_transitions stamped from preset_transitions (already exists!)
  • client_agents: academy × agent_template + mode (off/hawkeye/self_drive) + brain overrides
  • academy can tweak THEIR copy without touching the template
→
training per preset
  • agent_lessons.preset real column (backfill 'free_trial')
  • lessons key on agent_template, not a hardcoded name
  • readers load: academy lessons + general lessons WHERE preset matches
  • /consolidate-lessons runs per preset - free-trial craft never leaks into paid-assessment academies
Why "template + instance" and not just config flags: a preset is authored once and stamped many times. When BAM improves the free-trial playbook, new academies get the improved stamp while existing academies keep their tuned copy. Same pattern as the brain sections today (default in template, override per academy) - just promoted one level up.
how we get there without breaking BAM GTA ↓↓
D THE PATH — 4 phases, no big-bang rewrite
each phase ships value on its own
Phase 0done ✓
Foundation (already live). Stage registry, transition graph + router, automations engine, brain-as-data, preset-tagged lessons, Hawkeye training loop. This is why the answer is "3 pieces away," not "rewrite."
Phase 1open the roles
Unweld the stage list + move agent prose into data. Widen stage_role from a 7-value enum to open text validated against the registry, and move the free-trial prompt defaults from code constants into seeded rows. New stage names become possible; the free-trial "mission" becomes editable content. ⚠ enum→text touches live prod tables - needs its own careful migration + core review
Phase 2the preset noun
Create the template entities. pipeline_presets + preset_stages + preset_transitions + agent_templates; seed free_trial from today's exact model; replace the one-off seed function with apply_preset(academy, preset). You can now CREATE a second playbook and stamp a new academy with it in one step.
Phase 3generic workers
One engine, many agents. Collapse the 3 copy-pasted agent APIs into one detector/drafter parameterized by (academy, stage, agent_template). Hawkeye's tabs render from the academy's actual stages instead of hardcoded booking/confirm/closing. A 4th agent (or a 2-agent preset) is a data row, not a new API file.
Phase 4train per preset
Preset-scoped training. agent_lessons.preset column (backfill free_trial), readers filter general lessons by the academy's preset, /consolidate-lessons runs per preset, intake-gap mining feeds preset-specific onboarding questions. Every preset develops its own shared brain - free-trial wisdom never contaminates a no-trial academy.

Order matters: 1 before 2 (templates need open roles), 2 before 3 (the engine needs templates to read), 4 can start any time after 2.

decisions only Zoran can make ↓↓
E OPEN DECISIONS — product calls, not tech calls
these shape the schema - decide before Phase 2
1 · Preset per academy, or per OFFER? Today one academy = one pipeline. But offers already exist as their own thing (Training is one offer type). Can one academy run free-trial for youth training AND paid-assessment for elite? That decides where the preset hangs.
2 · Who authors presets? BAM-only templates, or can big academies fork their own? Recommend: BAM-only until preset #3 exists. Forks multiply the training surfaces.
3 · How much can an academy customize their copy? Rename stages? Re-wire transitions? Swap an agent for a human? Every degree of freedom here is support burden later. Recommend: transitions + agent mode yes, stage structure no.
4 · What's preset #2, really? Designing the abstraction against ONE real second case (paid assessment? drop-in classes? memberships-direct?) beats designing in the abstract. Pick the next academy type you actually want to sign.
