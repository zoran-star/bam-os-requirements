Move all automations into the portal

Stop building automations inside GoHighLevel one sub-account at a time. The portal owns the logic — triggers, timing, decisions — and GHL drops to being a pipe (CRM mirror + a way to send a text). Email moves to Resend so we own the whole message.

The whole idea in one sentence

We already built this pattern — the booking + confirm sales agents run entirely in the portal and only touch GHL to send. "All automations in the portal" = pointing that same pattern at everything else, and swapping GHL's weak email for Resend.

01

Why bother

GHL workflows today
  • Set up per sub-account by hand - 40+ academies = 40× the work
  • Change one thing → redo it everywhere
  • Can't branch on portal data (Stripe status, our DB, AI judgment)
  • Weak email - no real templates, no delivery visibility
  • Black box when something misfires
Portal-owned automations
  • Write once → every academy gets it
  • Branch on anything we know (DB, Stripe, AI, history)
  • Version-controlled, observable, testable
  • Resend email: our domain, templates, open/bounce logs
  • Human-approval option on any step (like Hawkeye)

The reusable pattern (every migrated automation)

1

Trigger

A cron tick, a GHL inbound webhook, or a portal event (Stripe failed, trial booked, lead went quiet).

▸ vercel crons · inbound-webhook · stripe/webhook · an automation_events table
2

Decide — in code

Who qualifies, which step they're on, what to say. Branch on DB + Stripe + AI. This is the part GHL can't do well.

▸ per-automation module (like the ghost detector)
3

Schedule — one generic queue

Drop a "send THIS to THEM at time T" row. Respect quiet hours. Carry a dedupe key so we never double-fire.

▸ NEW automation_jobs table (generalizes agent_followups + mass_send_*)
4

Send — through the pipe

One unified sender: SMS via the academy's GHL number, email via Resend. Skip DND / unsubscribed.

▸ NEW _send.js → ghl/_core sendSms + NEW _email.js (Resend)
5

Log + (optional) approve

Record every send + delivery result. Any step can route to a Hawkeye-style approval queue first.

▸ message_log + email_events; reuse the approval-card pattern
05

What's missing to generalize it

Today each automation has its own table (agent_followups, agent_ready_replies, agent_confirm_replies, mass_send_*). To scale, we extract the shared spine.

① Generic job queue

automation_jobs: client_id, contact_id, automation, step, channel, template, payload, run_after, status, dedupe_key, attempts. One worker cron drains it. Replaces N per-feature tables.

② Events table

automation_events: a normalized log of "X happened" (trial booked, payment failed, lead quiet) that automations subscribe to — instead of each one re-deriving state from GHL.

③ Resend wrapper

api/_email.js — a thin sendEmail() paralleling sendSms(). Plus a unified _send.js that picks SMS-or-email + checks quiet hours + suppression.

④ Template store

automation_templates: subject + body with {{vars}}, per channel, editable in the portal (like the agent Brain editor). Render with simple var substitution.

♻️
Mostly assembly, not invention. The job queue generalizes mass_send_jobs; quiet hours (_quiet.js), the send pipe (ghl/_core sendSms), and the approval-card UI already exist. The genuinely new bits are Resend email + the generic queue/events tables.
06

Resend — owning email

What we build
  • api/_email.js → sendEmail({to,from,subject,html,replyTo,tags})
  • Templates in automation_templates (or a templates/ dir)
  • api/resend/webhook.js → write delivered / opened / bounced / complained to email_events
  • A suppression list (hard bounces + unsubscribes) the sender checks
  • Swap GHL type:"Email" sends → _email.js
☑ What YOU need to do (manual)
  • Confirm / open the Resend account (already used for password resets)
  • Add + verify a sending domain (e.g. mail.byanymeansbusiness.com)
  • Paste Resend's DNS records at the registrar: SPF (TXT), DKIM (Resend's CNAMEs), DMARC (TXT)
  • Add RESEND_API_KEY to Vercel env
  • Decide the From identity (one BAM domain now; per-academy domains later)
📮
Deliverability is a warmup game. A fresh domain can't blast day one. Start with low-volume transactional (confirmations, receipts), then ramp drips. DMARC + DKIM are non-negotiable or we land in spam.

Phased roadmap

Phase 0 · Foundations~1 build cycle

automation_jobs + worker cron · automation_events · api/_email.js (Resend) + _send.js unified sender · automation_templates · Resend domain verified. Depends on: your DNS/Resend checklist above.

Phase 1 · Ghosted / nurture (the template)~1 build cycle

Build nurture on the new spine. Run in parallel with GHL (no double-send — see risks). Prove it, then turn off ghosted_workflow per academy. Output = the copy-paste template for everything after.

Phase 2+ · Migrate the rest1 per cycle

In leverage order: failed-payment dunning (money) → onboarding drips (Resend-heavy) → lost-lead nurture → booked-trial confirmation (fold into confirm agent) → email blasts. Each one retires its GHL workflow when it goes live.

🔌
SMS stays on GHL for now. Keep sending texts through the academy's GHL number (good deliverability, already wired). Re-evaluate Twilio only if GHL sending becomes a limit — not part of this plan.
09

Risks & open questions

⚠️

Double-send vs GHL native triggers

We've hit this before with ghosted. Each migration must turn OFF the matching GHL workflow (or gate it) the moment the portal version goes live. Track which workflow each automation replaces.

📮

Email deliverability / warmup

New domain → ramp slowly, transactional first. Needs SPF+DKIM+DMARC and a suppression list, or drips tank inbox placement.

🔁

Idempotency

Crons retry. Every job needs a dedupe_key and a "claim before send" status so a contact never gets the same touch twice.

🔭

Observability

One place to see "what got sent to whom, when, did it land." The message_log + email_events + a small staff view.

📱

SMS provider — later call

Staying on GHL for SMS now. If we ever want full independence, Twilio is the path (numbers, A2P 10DLC registration) — out of scope today.

🌐

From-domain strategy

One BAM domain for all academies now (simple) vs per-academy domains later (best branding + isolation). Start single, design the table to allow per-academy.

🎯 Goal

Every automation runs in the portal: write once, every academy gets it, branch on anything we know, own the email. GHL becomes the CRM mirror + the SMS pipe — nothing more. First proof: portal-owned nurture, replacing the GHL ghosted workflow.
