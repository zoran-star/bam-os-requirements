01The big picture

Most BAM GTA athletes already live in CoachIQ (the credits + scheduling engine). The goal was to put FullControl on top — running billing, CRM and marketing — without ripping out the tool the academy already runs every day.

The whole connection comes down to one idea:

FullControl owns the Stripe subscription. CoachIQ keeps owning credits & scheduling. The two are linked by the Stripe subscription ID.

Because FullControl creates and owns the subscription, the portal's billing buttons — pause, cancel, change plan, refund — actually work. CoachIQ, meanwhile, watches that same subscription ID so it knows when to refresh an athlete's credits. Neither system has to be the other's database; they just share one reference.

Why this matters BAM builds no custom credit bridge per academy. CoachIQ already knows how to watch an external subscription ID and grant credits off it. That means this same pattern can be sold to any academy already on CoachIQ — keep CoachIQ, add FullControl.

02Who owns what

The cleanest way to hold the model in your head: draw a hard line down the middle. Money on one side, the athlete experience on the other.

FullControl portal
owns the money
  • Creates & owns the Stripe subscription
  • Billing actions: pause · cancel · change plan · refund
  • The CRM, marketing, ad spend, support tickets
  • Surfaces the subscription ID for CoachIQ to track
  • Tells CoachIQ when to grant a product / credits
CoachIQ
owns the experience
  • Credits — granting & redeeming on booking
  • Scheduling — classes, calendars, check-ins
  • The athlete app the parent actually uses (white-labeled)
  • Product / program access per athlete
  • Watches the subscription ID to refresh credits on renewal
Sacred rule Payment only happens on the FullControl side. Never put a payment / Products connection on a CoachIQ form — if CoachIQ takes the payment, CoachIQ owns the subscription and the billing buttons stop working.

03Webhooks, both ways

The two systems talk over webhooks in both directions. CoachIQ tells FullControl when something happens to a user; FullControl tells CoachIQ when to grant a product or credits. Nothing is polled on a timer — each side just pokes the other when there's news.

CoachIQ ──▶ FullControl
CoachIQ automations with a "Send to External Webhook" action. CoachIQ is the trigger; FullControl listens.
New user signed upParent made their CoachIQ account → FullControl matches them to a paying member and grants the product.
Tag added to a userUsed during member import: tag a member in CoachIQ → FullControl captures their CoachIQ user ID and links it.
FullControl ──▶ CoachIQ
FullControl fires a CoachIQ automation by its "Incoming Webhook" trigger. FullControl is the trigger; CoachIQ acts.
Grant the productA member paid in the portal → tell CoachIQ to add the product + access + starting credits (no payment taken).
Identify the athleteEvery message names the exact CoachIQ user to act on, so the right athlete gets the credits.
The one gotcha CoachIQ can only act on a specific CoachIQ user ID — not an email, not a profile link. That's why FullControl has to store each member's CoachIQ user ID first. Capturing that ID is exactly what the inbound "new user" and "tag added" webhooks are for (see Member import).

04Automations to create in CoachIQ

Three CoachIQ automations make the connection work. The academy (or BAM systems) builds these once inside CoachIQ. Two of them talk out to FullControl; one listens in from FullControl.

ANew User → Send to External Webhookoutbound
TriggerNew User — a parent creates their account on the academy's group login page
ActionSend to External Webhook → tells FullControl the new user's ID + email
ResultFullControl matches that email to a paying member, then fires automation C to grant their product
BUser Tag Added → Send to External Webhookoutbound
TriggerTag added to a user — a staff member tags an existing athlete in CoachIQ
ActionSend to External Webhook → tells FullControl that user's ID + email
ResultFullControl links that CoachIQ user ID onto the matching member row (the import "listening" step)
CAdd a Product Purchaseinbound · one per plan
TriggerIncoming Webhook — fired by FullControl when a member pays
ActionAdd a Product Purchase → grants the product, program access, and the cycle's starter credits — without taking payment (they already paid in the portal)
Storesthe FullControl subscription ID on the product, so CoachIQ can track renewals and refresh credits
One per plan Automation C is built once per product / plan (e.g. 1×/wk, 2×/wk, 3×/wk, Unlimited). FullControl picks the right one based on what the member actually bought, so each plan grants its own product and credit amount.

Onboarding an existing roster means matching each athlete in the portal to their existing CoachIQ user. The import wizard does this with a live "listening session" — it watches matches arrive in real time instead of making you copy IDs by hand.

How the CoachIQ step listens

The portal needs each member's CoachIQ user ID. There's no CoachIQ page that simply shows it, so the wizard gathers it three ways and ticks members off the list live as their ID lands:

How a match arrivesWhat happens
Auto, from StripeFor subscriptions CoachIQ originally created, CoachIQ stamped the user ID into the subscription's metadata. The wizard reads it off automatically — no action needed.
Live, by taggingFor the rest, a staff member adds a tag to the athlete in CoachIQ. Automation B fires, FullControl captures that user's ID, and the member ticks off on screen within seconds.
Manual fallbackPaste a user ID by hand, or mark a member "not on CoachIQ" so they're skipped cleanly.
Never blocks A member with no CoachIQ match doesn't stop the import — you can still promote them. The listening screen just keeps watching, and any tag that doesn't match a known member is flagged so you can see it.

06The sub ID link

The subscription ID is the single thread tying the two systems together. Understanding it explains the whole model.

How the sub ID flows
1
FullControl creates the subscription
When a member pays, the portal creates a Stripe subscription it owns → it gets a subscription ID.
2
FullControl hands the ID to CoachIQ
The "grant product" webhook carries the subscription ID, and CoachIQ stores it on the athlete's product.
3
CoachIQ watches that subscription
CoachIQ tracks the subscription's renewal date and refreshes the athlete's credits each cycle, off that ID.

Because credits are pinned to the subscription ID and not to who created the payment, FullControl can own the money while CoachIQ keeps doing what it's good at. Pause the subscription in the portal and renewals stop; the credit cycle naturally stops with it.

Plan changes need a re-link If a member changes plan, they may end up on a new product (and sometimes a new subscription ID). CoachIQ has to be pointed at the new product / ID — see the day-to-day guide below. This re-link is the academy's normal CoachIQ workflow, not a BAM engineering task.

08Running it: the guide

Day-to-day, almost everything starts in the FullControl portal. The rule of thumb: change the money in FullControl, then make CoachIQ match. Here's what each common action means across both systems.

Where to do each action

You want to…Do it inWhat CoachIQ needs after
Sign up a new memberFullControl (payment) → parent makes CoachIQ accountNothing manual — automations A + C handle the grant
Change a planFullControl (change plan)Re-link: remove the old product, add the new one, paste the updated sub ID, set credits
Pause a membershipFullControl (pause)Nothing — renewals stop, so the credit cycle stops
Cancel a membershipFullControl (cancel)Remove / revoke the product if the academy wants access to end immediately
RefundFullControl (refund)Nothing billing-side; adjust credits only if the academy chooses to

What "action items" are

Most of the time the two systems sync themselves through the webhooks. An action item is the short list of things a human still has to do by hand — and they almost always live on the CoachIQ side, after a change in FullControl:

  • Re-link after a plan change — point CoachIQ at the member's new product + sub ID. This is the big one.
  • Confirm a tricky import match — a member whose CoachIQ ID didn't auto-arrive (tag them, or mark "not on CoachIQ").
  • Revoke access on cancel — only if the academy wants the athlete locked out the moment they cancel, rather than at period end.
  • Re-collect a card — for the rare member migrated without a reusable card on file.
