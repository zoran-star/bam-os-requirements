Scale & Security Audit

Round-2 audit focused on (1) what breaks past 50 clients and (2) actual exploitable security issues. The earlier infrastructure map covered architecture + dead code; this one looks at performance + risk.

TL;DR — what to fix this week

  • 3 CRITICAL security issues: hardcoded anon key fallback, email enumeration on signup, Notion endpoint reads arbitrary pages with no auth.
  • 3 HIGH scale blockers: per-client Stripe lookup on every Clients tab load, unbounded ticket queries, missing indexes on hot foreign keys.
  • 2 HIGH security: client-portal session leak, internal messages can reach client view.
  • Rest are medium/low and can be batched.

📈 Scalability (50+ clients)8 findings

Today: 25 clients. These all get worse linearly or quadratically as that number grows.

S-1
Per-client Stripe revenue lookup
High
api/clients.js:607-612 · Promise.all(rows.map(getStripeRevenue))
Every time anyone opens the Clients tab, the API fires one Stripe API call per client to get live MRR. With 50 clients that's 50 calls; at 200 clients it's 200 calls every page load.
Clients tab takes longer to load each time you add a client. Eventually hits Stripe rate limits or just becomes unusable.
Cache MRR in Supabase (refreshed nightly or via Stripe webhook), or batch with one ?expand=subscriptions Stripe call instead of N customer fetches.
S-2
Unbounded ticket queries (no LIMIT)
High
api/marketing.js:225, 231, 589, 600 · sb('marketing_tickets?select=*…')
Marketing + content ticket fetches have no limit= clause. Right now there are dozens of rows total. At 50 clients each averaging ~10 tickets/year that's 500/year per table. Both will balloon.
As tickets accumulate, every Marketing/Content tab load downloads the entire history. Browser memory + Vercel payload size go up forever.
Add ?limit=50&offset=…&order=submitted_at.desc. Add a "Load older" button in the UI. Same for content tickets.
S-3
Missing indexes on hot foreign keys
High
Supabase tables: tickets, marketing_tickets, content_tickets, client_notes
Only clients(scaling_manager_id) has an index. Every other client_id FK is unindexed. Every query like "tickets for this client" does a full table scan.
Per-client detail page Activity tab and Marketing tab get slower as ticket count grows. Postgres falls off a cliff around ~10k rows on unindexed scans.
Add B-tree indexes on every *_tickets.client_id, client_notes.client_id, and any created_at DESC we filter on. Single migration, ~10 lines of SQL.
S-4
No pagination on Clients list
Med
src/views/ClientsCombinedView.jsx:72 · supabase.from("clients").select("*")
All clients loaded into a single React state array. Search/filter/sort happen client-side. Fine for 25, sluggish at 100, painful at 300.
Typing in the search box re-renders all rows. Initial paint takes longer. Eventually the list itself becomes the slowest thing on screen.
Move filtering/sorting/searching server-side (Supabase .ilike() + .range()). Or virtualize the list with react-window. Cheap win when needed.
S-5
Per-DM Slack message fetches on Dashboard
Med
src/views/DashboardView.jsx:72-88 · dms.slice(0, 5).map(async ch => fetchMessages(ch.id))
Every Dashboard load fetches the last messages for 5 DMs in parallel. Slack rate limit is ~50 req/min for that endpoint.
Multiple staff hitting the dashboard simultaneously could throttle. Adds 5 round-trips to every dashboard load.
Cache last-message-per-DM in Supabase via a Slack Events webhook subscription. Or just don't show the message preview on Dashboard — Communication tab covers it.
S-6
File uploads use Promise.all on unlimited batch
Med
src/views/ContentView.jsx:293-305 · incoming.map(file => supabase.storage…upload(…))
If a content team member drops 20 raw video files at once, all 20 start uploading at the same time. Browser network can choke.
Bad UX on flaky wifi, slows down concurrent users sharing bandwidth.
Limit concurrency to 3 simultaneous uploads with a simple queue. Show progress per file.
S-7
Communication tab polls every 10s
Low
src/views/CommunicationView.jsx:386-399 · setInterval(fetchMessages, 10000)
Polling instead of subscribing. One staff person leaving the tab open all day = 8,640 polls/day. Multiply by team size.
Wastes API calls and Slack rate limit budget. Real-time UX is also worse (10s lag).
Use Supabase Realtime channel subscriptions for message updates. Already in the round-3 backlog.
S-8
GHL cache hit rate could be higher
Low
api/ghl.js:10-32 · 5 min TTL
GHL data (locations, contacts, pipelines) is cached but only for 5 minutes. Many of these change daily, not by the minute.
More GHL API calls than necessary. GHL has rate limits.
Bump TTL to 60 min for location list. Keep 5 min for conversations.

🔒 Security9 findings

Real exploitation paths against THIS codebase, not theoretical hardening. Severity tracks impact + ease of exploitation.

SEC-1
Hardcoded Supabase anon key in client code
Critical
src/lib/supabase.js:4 · fallback when env var missing
The Supabase anon JWT is hardcoded as a fallback if VITE_SUPABASE_ANON_KEY isn't set. Anon key is meant to be public, but having it baked in means: (a) we can't easily rotate it without a code change, (b) if Vercel env var ever fails to set, we silently fall back to a literal hardcoded key that may differ from what RLS policies were written for.
Key rotation becomes painful. If RLS isn't fully locked down (and it mostly relies on API gating, see SEC-8), anyone with the URL + key can hit PostgREST directly.
Remove fallback. throw new Error("VITE_SUPABASE_ANON_KEY missing") if not set. Verify all tables have actual RLS policies, not just auth.role() = 'authenticated'.
SEC-2
Email enumeration on public signup
Critical
api/clients.js:169-172 · public signup error messages
Signup returns distinct errors for "email already exists" vs "invalid format" vs other failures. Attacker can probe POST /api/clients with any email and learn whether it's already in the system.
Anyone can harvest the full list of registered client emails by spraying. Useful for spear-phishing your clients. No rate limit makes this trivial.
Always return the same generic success ("If that's a new account, an invite is on its way") regardless of whether the email exists. Send the email async. Add rate limiting (10/IP/day).
SEC-3
Notion endpoint reads any pageId, no auth
Critical
api/notion/query.js:588-657 · type=sop_content accepts user-supplied pageId
The endpoint has no auth check. For type=sop_content, it accepts an arbitrary pageId from the request body and fetches that Notion page using the integration key. An attacker can pass any Notion page ID the integration has access to — including internal notes, financials, strategy docs.
Anyone with the public URL can read any Notion page in your workspace the BAM integration has been shared with. Could include sensitive business info.
Add staff auth check at the top of the handler. Whitelist allowed pageIds to the constants (SOP_PAGES, CLIENT_PROFILES_PAGE, WAREHOUSE_*). Reject any pageId not in the whitelist.
SEC-4
Client portal session — token-to-client mapping
High
api/tickets.js:126-140 + api/marketing.js:67-72
Client portal resolves client_id from the user's Supabase auth_user_id once at request start, then trusts it for all subsequent ops. If a single auth_user is ever linked to multiple clients (current schema doesn't prevent it), or if a token leaks, the attacker reads/writes tickets across clients.
Lateral movement: one compromised client login → access to other clients' tickets if the data model allows multi-client auth_user.
Add a unique constraint: clients.auth_user_id already nullable; make it UNIQUE. Re-verify token freshness on each mutating op. Use shorter TTLs for client portal tokens.
SEC-5
Internal messages can reach client view
High
api/marketing.js:90-93 (stripInternalMessages) + line 363-430 (request-content-revision)
Marketing ticket messages have an internal: true flag. stripInternalMessages() filters them out, but it's only called in some code paths. If a client requests with the wrong scope param or hits a path that uses enrichWithClient() (which doesn't strip), they see staff-only handoff notes.
Clients can read internal staff conversation about their tickets — work-in-progress feedback, internal blame, staff handoff notes.
Strip internal messages in every client-facing response path. Add an explicit test that runs the client endpoint and asserts no message with internal:true is returned.
SEC-6
Ticket files bucket is public
High
Supabase Storage · ticket-files bucket · per round-3 backlog
Files uploaded to support tickets (videos, screenshots, raw assets) live in a public bucket. URLs are guessable patterns. Anyone who knows or guesses a URL can download anyone's files.
A client's raw video assets, error screenshots that may contain PII, or marketing creatives can be downloaded by anyone with the URL pattern.
Switch the bucket to private. Generate signed URLs server-side per request with short TTLs. Listed in round-3 backlog (deferred from MVP).
SEC-7
No rate limit on public signup
Med
api/clients.js:127-192 · public signup path
Anyone can hit POST /api/clients with no auth in a loop. Each call creates a Supabase auth user + sends an invite email via Resend.
Attacker can spam create thousands of bogus client accounts, exhaust Resend email quota (100/day on free), pollute auth.users + clients tables.
Add IP-based rate limiting (e.g., Vercel Edge middleware: 10 signups/IP/day). Consider Cloudflare Turnstile or hCaptcha on the form.
SEC-8
RLS bypassed by service key — no defense in depth
Med
All api/*.js · all queries use SUPABASE_SERVICE_KEY
Every API route uses the service role key, which bypasses RLS entirely. All role/permission gating is in API code. If a future API bug forgets a role check, the database has no second line of defense.
One missed if (!isAdmin) return 403 in a future PR = full data exposure. RLS is your fallback; right now it's disabled at runtime.
For client-portal-facing endpoints, switch to the anon key + user JWT so RLS does the gating. Keep service key for staff-only admin actions. Two-tier model.
SEC-9
Sensitive data potentially in server logs
Med
api/clients.js:592, api/tickets.js:361, api/marketing.js:187 · console.error(err)
Several catch blocks log err or err.message without sanitizing. Supabase error messages can echo back the offending row data (including emails, tokens) or the request body.
Vercel logs may contain client emails or auth tokens. If a Vercel session is ever compromised, those logs become a goldmine.
Log err.message only (not err), and never log request bodies in catch blocks. Strip Authorization headers from any log object.
SEC-10
Notion endpoint quota exhaustion (DOS)
Med
api/notion/query.js · no auth at all
Beyond SEC-3 (reading arbitrary pages), the endpoint can be hit in a loop with no auth, no rate limit. Each call burns Notion API quota.
Attacker can DOS your Notion integration by exhausting daily quota. Knowledge Base + Tasks tabs go blank until quota resets.
Same fix as SEC-3 (add staff auth check). Rate limit doesn't hurt either.
SEC-11
No CSRF protection on staff mutations
Med
All POST/PATCH endpoints · trust Bearer token only
Mutations rely on Bearer tokens, not CSRF tokens. Supabase tokens live in localStorage so they'd need an XSS to leak — but there's no defense if XSS ever happens.
Lower-tier risk because no dangerouslySetInnerHTML usage in the codebase. But CSRF is still standard hygiene.
Standard CSRF tokens on mutating routes, or use SameSite=Strict cookies for the staff session.
SEC-12
Hardcoded GHL location IDs / Notion IDs (already covered)
Low
api/ghl.js · api/notion/query.js
Constants in code rather than DB. Not a security issue per se, but if the integration's permissions ever expand, the whitelist (per SEC-3 fix) becomes load-bearing.
Configuration drift if Notion shares more pages with the BAM integration than the code allows for.
Document the integration's allowed pages list. Audit Notion permissions quarterly.
SEC-13
No CSP / security headers
Low
vercel.json — no headers config
No Content-Security-Policy, X-Frame-Options, Strict-Transport-Security, etc.
If an XSS ever happens, no CSP defense. Site could be iframe'd by phishing sites.
Add headers block to vercel.json with standard hardening.
