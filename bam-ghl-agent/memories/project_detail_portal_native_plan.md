## The key architectural fact (why the order is what it is)
GHL messaging is **contact-centric**: outbound needs a real GHL `contactId` (`api/ghl/send-message.js:337`), inbound webhooks only fire for GHL contacts. The built portal-native contact path (`api/_contacts.js:53` `resolveOrMintPortalContact`) **makes NO GHL call** - it mints a uuid used as both `contacts.id` and `ghl_contact_id`. So portal-native contacts + GHL messaging are **incompatible** without net-new dual-write code.

**Resolution = don't fight it.** While GHL is the phone line, contacts stay `contact_provider='ghl'` (GHL master, portal mirrors via `ghl_contacts`). Academy never logs into GHL anyway (V1.5), so it already feels fully portal-native. **The contacts flip to `contact_provider='portal'` is the LAST step, gated on the Twilio cutover** (once `messaging_provider='twilio'`, GHL is no longer the phone line, so contacts can leave safely). Zero throwaway bridge code.
