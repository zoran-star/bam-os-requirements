---
name: Separate data points from features in review docs
description: Onboarding review docs must clearly separate what data to collect vs what features to build
type: feedback
---

In the onboarding review docs, clearly separate **data points to collect during onboarding** from **features to build in the product**.

**Why:** Zoran flagged that the Scheduling section mixed both together. It's fine to include both, but they need to be visually separated so the team knows what's an onboarding input vs a product feature spec.

**How to apply:** In future review docs, use separate subsections or tags to distinguish:
- **Data point** = something the owner provides during setup (goes in the Onboarding Data Points DB)
- **Feature** = something the platform does (goes in the business requirements / feature specs)

Both are valuable, but they get tracked in different places. When saving to Notion, only add actual data points to the Onboarding Data Points database. Features go into the business requirements pages.
